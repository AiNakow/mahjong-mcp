# 推荐和评分策略改进方案

本文记录已经落地的麻将决策逻辑与解释系统改进。

## 1. 决策模式调整

目标：一向听、高打点手牌在面对威胁时更偏向进攻，而不是固定进入 `balance`。

当前规则：

```ts
function chooseStrategyMode(state: GameState, shanten: number): StrategyMode {
  const hasRiichiThreat = state.opponents.some((opponent) => opponent.riichi);
  if (!hasRiichiThreat) return "attack";
  if (shanten >= 2) return "defense";
  if (shanten === 0) return "push";
  if (shanten === 1 && isHighValueHand(state)) return "push";
  return "balance";
}
```

高打点手牌判定：

```ts
function isHighValueHand(state: GameState): boolean {
  const doraTiles = state.doraIndicators.map(nextDoraTile);
  const selfTiles = [...state.self.hand, ...(state.lastDraw ? [state.lastDraw] : [])];
  const doraCount = selfTiles.filter((tile) => doraTiles.includes(tile)).length;

  return doraCount >= 2 || state.averageWaitPoints >= 7700;
}
```

目前代码中 `GameState` 标准类型尚未声明 `averageWaitPoints`，实现会兼容读取该可选字段；若没有该字段，则只按宝牌数量判定。

## 2. 防守权重动态调整

目标：高打点手牌时降低过度防守对牌效和打点的压制。

当前模式权重：

```ts
if (mode === "balance") {
  defenseWeight = highValueHand ? 2 : 5;
}

if (mode === "push") {
  defenseWeight = 2;
  valueWeight *= 1.15;
}
```

即一向听高打点面对立直会进入 `push`，保留一定防守风险评估，但不会像 `defense` 模式一样强行弃和。

## 3. 防守余力 / 后续防守空间

目标：避免切掉唯一安全牌，导致“攻不了也防不了”。

当前防守综合分：

```text
defenseScore =
  immediateSafetyMinusDanger(discard)
  + defenseReserveScore(remainingTiles)
  - futureDefensePenalty(safeTileCountAfterDiscard)
```

`defenseReserveScore` 对切牌后的剩余手牌计分：

```ts
function defenseReserveTileScore(tile: Tile): number {
  if (isGenbutsu(tile)) return 100;
  if (isSujiTerminal(tile)) return 45; // 1/9 筋
  if (isSuji28(tile)) return 35;       // 2/8 筋
  if (isSuji456(tile)) return 20;      // 4/5/6 筋
  if (isSuji37(tile)) return 10;       // 3/7 筋
  return 0;
}
```

“较可靠防守牌”当前按安全分 `>= 35` 计入，也就是现物、`1/9` 筋和 `2/8` 筋。`3/7` 筋不会被算作较可靠防守资源。

后续防守不足惩罚：

```ts
function futureDefensePenalty(safeTileCountAfterDiscard: number): number {
  if (safeTileCountAfterDiscard === 0) return 220;
  if (safeTileCountAfterDiscard === 1) return 120;
  if (safeTileCountAfterDiscard === 2) return 50;
  return 0;
}
```

## 4. 筋牌安全度分级

目标：弱筋不要被高估，尤其是 `3/7` 筋。

当前筋牌安全度按以下顺序逐渐降低：

```ts
function sujiSafetyScore(tile: Tile, isGenbutsu: boolean): number {
  if (isGenbutsu) return 100;
  if (isSujiTerminal(tile)) return 45; // 1/9
  if (isSuji28(tile)) return 35;       // 2/8
  if (isSuji456(tile)) return 20;      // 4/5/6
  if (isSuji37(tile)) return 10;       // 3/7
  return 0;
}
```

这样 `3m/7m` 这类弱筋只提供很小的安全补正，不会被误判为强安全牌。

## 5. 推荐理由系统改进

目标：

- “理由”只显示支持推荐动作的非负面理由。
- 高打点理由优先级提升。
- 负面理由单独展示为“注意”。

当前解释层流程：

```ts
const reasons = normalizeReasonPriorities(best.reasons)
  .filter((reason) => reason.polarity !== "negative")
  .sort((a, b) => b.priority - a.priority)
  .slice(0, 4);

const warnings = best.reasons.filter((reason) => reason.polarity === "negative");
```

高打点优先级：

```ts
if (points >= 12000) priority = 92;
else if (points >= 7700) priority = 86;
else priority = Math.max(priority, 64);
```

## 6. 比较型解释

目标：解释“为什么选这个动作而不是另一个”。

当前已支持两类比较：

- 进张比较：推荐候选进张数高于第二候选时说明进张优势。
- 打点路线比较：推荐候选打点路线优于第二候选时说明路线差异。
- 防守比较：推荐候选进张较少但防守分更高时，说明牺牲部分进张换取风险下降和后续防守资源。

防守比较示例：

```text
相比切 7s，切 4z 牺牲部分进张但显著降低对威胁者风险并保留后续防守资源。
```

## 总结

| 模块 | 改动目的 | 当前修改 |
| --- | --- | --- |
| 决策模式 | 一向听高打点偏进攻 | `shanten=1 && highValueHand -> push` |
| 防守权重 | 避免高打点手牌过度保守 | `balance` 模式动态防守权重：高打点 `2`，普通 `5` |
| 防守余力 | 避免切掉唯一安全牌 | 增加 `defenseReserveScore` 和 `futureDefensePenalty` |
| 筋牌权重 | 弱筋不要被高估 | `1/9=45`、`2/8=35`、`4/5/6=20`、`3/7=10` |
| 推荐理由 | 解释更聚焦 | 正面理由进“理由”，负面理由进“注意”；高打点理由提优先级 |
| 比较型解释 | 增强可理解性 | 支持进张、打点路线、防守资源对比 |
