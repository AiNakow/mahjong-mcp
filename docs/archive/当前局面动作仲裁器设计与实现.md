# 当前局面动作仲裁器设计与实现

## 1. 背景和目标

当前项目已经具备较完整的切牌推荐能力：

- `chooseAction(state)` 可以在自家摸牌后的 `3n+2` 手牌中推荐切牌。
- `evaluateNanikiru` 已经整合牌理、打点路线、防守、顺位和 EV 二次仲裁。
- `evaluateRiichiJudgment` 已经能对切后听牌候选判断是否适合立直。
- `calculateAgariScore` 已经能完成自摸、荣和、役种、番符和点数计算。

当前 `chooseAction` 已经从早期“何切入口”升级为统一动作推荐入口。它会先生成合法动作集合，再把自摸、荣和、切牌、立直、副露、杠和不鸣转换为统一候选进行评分和仲裁。本文保留原始设计目标，同时记录当前实现状态和剩余边界。

设计目标是把 `chooseAction` 作为统一动作决策入口：

```ts
chooseAction(state: GameState, options?: ChooseActionOptions): ActionDecision
```

它应该根据当前局面自动生成所有合法动作，分别评分，再返回推荐动作、候选比较、解释和必要的 EV 明细。

目标动作范围：

- 自摸和牌：`tsumo`
- 荣和：`ron`
- 普通切牌：`discard`
- 立直切牌：`riichi`
- 吃：`chi`
- 碰：`pon`
- 大明杠：`minkan`
- 暗杠：`ankan`
- 加杠：`kakan`
- 跳过：`pass`

## 2. 总体架构

推荐将动作决策拆成四层：

```text
GameState
  -> determineDecisionPhase
  -> generateLegalActions
  -> evaluateAction / evaluateActionGroup
  -> arbitrateActions
  -> ActionDecision
```

职责划分：

| 层 | 职责 |
| --- | --- |
| 决策阶段识别 | 判断当前是自家摸牌、他家打牌，还是副露后需切牌 |
| 合法动作生成 | 只负责规则合法性，不做策略判断 |
| 动作评分 | 对每个合法动作计算进攻、防守、打点、顺位、EV 和解释理由 |
| 动作仲裁 | 在不同动作类型之间排序，决定最终推荐 |
| 解释渲染 | 基于候选理由输出人类可读解释，不重新计算策略 |

推荐新增模块：

```text
src/strategy/
  legal-actions.ts
  evaluate-action.ts
  action-arbitration.ts
  call-evaluation.ts
  kan-evaluation.ts
  agari-evaluation.ts
```

现有 `choose-action.ts` 保留为公开入口，但内部改为调用动作仲裁器。

## 当前实现状态

截至 2026-06-16，动作仲裁主干已完成第一轮落地：

- 已新增并接入 `DecisionPhase` / `DecisionAction` / `EvaluatedAction`，`chooseAction` 返回统一动作决策。
- `determineDecisionPhase` 支持显式 `phase`，并能处理 `self_draw`、`opponent_discard`、`after_call_discard`、`chankan` 和 `rinshan_draw`。
- `generateLegalActions` 已覆盖自家摸牌后的自摸、立直、暗杠、加杠和切牌，也覆盖他家打牌后的荣和、吃、碰、大明杠和不鸣。
- 切牌动作已从何切链路封装为 `discard` 候选，并按合法切牌集合过滤；副露后 `after_call_discard` 阶段只生成可切牌动作。
- `evaluateAgariActions` 只消费合法和牌动作；有效自摸/荣和会优先短路，不再继续运行昂贵何切、副露、杠和 EV 分析。
- 立直候选已从切牌候选升级为真实 `riichi` 动作，并与普通切牌同台仲裁。
- 吃、碰和大明杠候选会应用副露后的模拟局面，再复用 open mode 何切评分生成带后续切牌的完整候选。
- 杠动作已接入统一候选，并挂接快速 EV 近似；立直后暗杠采用保守合法性，只允许自摸牌补成手内暗刻第四张且暗杠前后待牌集合不变。
- `pass` 已作为不鸣/不和候选参与仲裁；南四等终局场景下，若和牌后仍不能提升名次，可以推荐见逃继续追逆转。
- `applyDecisionAction` 支持切牌、立直切牌、不鸣、吃/碰及其后续切牌、杠动作登记和和牌终局标记。
- 已补充基础振听、临时振听、立直见逃振听、食替限制和自家摸牌阶段临时振听解除。
- `decide` CLI 已同步支持轻量参数和完整 JSON 局面；抢杠、岭上摸牌和副露后切牌等阶段建议使用完整 JSON。

当前剩余重点：

- 加强 `pass` 与副露/荣和候选之间的解释稳定性，尤其是终局见逃和不鸣理由。
- 补齐杠动作边界测试：对立直不明杠、终局领先不加杠、已经可和时优先和牌。
- 优化副露、立直计划和二层估值相关慢测，保持 `npm run test:fast` 适合日常回归。
- 后续再扩展完整对局循环、多人荣和结算、包牌、四杠散了和更精确的排名 EV。

## 3. 决策阶段

`GameState` 支持显式 `phase`，同时保留基于 `lastDraw` / `lastDiscard` 的兼容判断：

```ts
export type DecisionPhase =
  | "self_draw"
  | "opponent_discard"
  | "chankan"
  | "rinshan_draw"
  | "after_call_discard"
  | "unknown";
```

兼容判断规则：

```ts
function determineDecisionPhase(state: GameState): DecisionPhase {
  if (state.lastDraw && !state.lastDiscard) {
    return "self_draw";
  }
  if (state.lastDiscard && state.lastDiscard.playerIndex !== 0) {
    return "opponent_discard";
  }
  return "unknown";
}
```

当前建议优先在完整 JSON 中提供显式字段，避免 `lastDraw` 和 `lastDiscard` 组合语义不清：

```ts
phase:
  | "self_draw"
  | "opponent_discard"
  | "chankan"
  | "rinshan_draw"
  | "after_call_discard";
```

其中：

- `self_draw`：自家摸牌后，可自摸、立直、杠、切牌。
- `opponent_discard`：他家打牌后，可荣和、吃、碰、明杠、跳过。
- `after_call_discard`：副露成功后，必须从副露后的手牌中切一张。
- `chankan`：他家加杠后的抢杠响应窗口。
- `rinshan_draw`：自家杠后的岭上摸牌阶段。

## 4. 动作类型

当前 `src/core/action.ts` 已有基础动作类型：

```ts
export type Action =
  | { type: "discard"; tile: TileId }
  | { type: "riichi"; tile: TileId }
  | { type: "tsumo" }
  | { type: "ron" }
  | { type: "chi"; tiles: TileId[] }
  | { type: "pon"; tiles: TileId[] }
  | { type: "kan"; tiles: TileId[] }
  | { type: "pass" };
```

这个类型可以表达公开动作，但不足以表达副露后的推荐切牌。吃、碰、杠推荐必须包含“副露后切什么”，否则无法成为完整建议。

推荐新增策略层动作类型，不直接替换核心动作：

```ts
export type DecisionAction =
  | { type: "discard"; tile: TileId }
  | { type: "riichi"; tile: TileId }
  | { type: "tsumo" }
  | { type: "ron" }
  | { type: "chi"; tiles: TileId[]; calledTile: TileId; discard: TileId }
  | { type: "pon"; tiles: TileId[]; calledTile: TileId; discard: TileId }
  | { type: "minkan"; tiles: TileId[]; calledTile: TileId; discard: TileId }
  | { type: "ankan"; tiles: TileId[]; discard?: TileId }
  | { type: "kakan"; tiles: TileId[]; discard?: TileId }
  | { type: "pass" };
```

说明：

- `discard` 是副露后或自摸后的实际切牌。
- `chi` / `pon` / `minkan` 必须带 `discard`，因为鸣牌后仍需要切牌。
- `ankan` / `kakan` 后摸岭上牌，现实流程不是立即切牌；第一版可以只推荐杠动作，不强行给后续切牌。
- 对外 JSON 可继续保留 `action`，同时增加 `followUp` 或在动作里内联 `discard`。

## 5. 统一候选结构

新增统一动作候选：

```ts
export interface EvaluatedAction {
  action: DecisionAction;
  phase: DecisionPhase;
  legal: boolean;
  score: number;
  priority: number;
  category: "agari" | "discard" | "riichi" | "call" | "kan" | "pass";
  scoreBreakdown: {
    agari?: number;
    speed?: number;
    value?: number;
    defense?: number;
    placement?: number;
    ev?: number;
    callCost?: number;
    kanRisk?: number;
  };
  estimate?: RoundEstimate;
  reasons: StrategyReason[];
  warnings: StrategyReason[];
  source?: unknown;
}
```

`source` 用于保留原始评分结果，例如：

- 切牌动作对应 `EvaluatedNanikiruCandidate`
- 和牌动作对应 `AgariScoreResult`
- 副露动作对应副露后最佳切牌分析

动作仲裁只读取结构化字段，不反向解析解释文本。

## 6. 合法动作生成

### 6.1 自家摸牌后

`self_draw` 阶段生成：

- `tsumo`
- `riichi`
- `ankan`
- `kakan`
- `discard`

伪代码：

```ts
function generateSelfDrawActions(state: GameState): LegalAction[] {
  return [
    ...generateTsumoAction(state),
    ...generateRiichiActions(state),
    ...generateAnkanActions(state),
    ...generateKakanActions(state),
    ...generateDiscardActions(state),
  ];
}
```

### 6.2 他家打牌后

`opponent_discard` 阶段生成：

- `ron`
- `chi`
- `pon`
- `minkan`
- `pass`

伪代码：

```ts
function generateOpponentDiscardActions(state: GameState): LegalAction[] {
  return [
    ...generateRonAction(state),
    ...generateChiActions(state),
    ...generatePonActions(state),
    ...generateMinkanActions(state),
    { type: "pass" },
  ];
}
```

### 6.3 副露后切牌

吃、碰、大明杠候选不能只生成副露动作，还要枚举副露后的切牌：

```text
他家打出 X
  -> 枚举可吃/碰/明杠组合
  -> 应用副露，得到 open hand
  -> 对副露后手牌跑 analyzeHandText / evaluateNanikiru
  -> 每个“副露 + 切牌”成为一个完整候选
```

这样动作之间可以比较：

- 不鸣保持门清
- 碰役牌后切 `9m`
- 吃断幺后切 `1p`

## 7. 各动作评估

### 7.1 自摸 `tsumo`

合法性：

- `state.lastDraw` 存在。
- `self.hand + lastDraw` 能通过 `calculateAgariScore` 得到 `status: "scored"`。

评分：

- 第一版默认极高分，通常优先于其他动作。
- 分数可由点数、顺位收益、供托、本场加成构成。

建议评分：

```text
score = 100000
  + agariPoints / 10
  + placementBonus
```

解释：

- 自摸成立。
- 展示最佳役种、番符、点数。
- 若南四或接近终局，展示顺位变化。

### 7.2 荣和 `ron`

合法性：

- `state.lastDiscard` 存在。
- `self.hand + lastDiscard.tile` 能通过 `calculateAgariScore` 得到 `status: "scored"`。
- 非振听。

当前振听判断：

- 基础振听：当前所有待牌中任一张曾出现在自家河里，则不能荣和。
- 临时振听：对可荣和弃牌选择 `pass` 后进入临时振听，推进到自家摸牌阶段时解除。
- 立直见逃振听：立直后对可荣和弃牌选择 `pass` 后进入 `riichiFuriten`，后续自摸推进中保持。

评分：

```text
score = 100000
  + ronPoints / 10
  + placementBonus
```

说明：

- 默认推荐和牌。
- 终局特殊情况下可以允许见逃，但 MVP 不建议加入复杂例外。

### 7.3 普通切牌 `discard`

继续复用现有主链路：

- `analyzeHandText`
- `evaluateNanikiru`
- `applyEvDecision`
- `applyRiichiPlanDecision`

改造点：

- `discard` 候选不再直接作为 `chooseAction` 的唯一结果。
- 每个推荐候选转成 `EvaluatedAction`。

映射方式：

```ts
{
  action: { type: "discard", tile: candidate.discard },
  category: "discard",
  score: candidate.score,
  scoreBreakdown: candidate.scoreBreakdown,
  reasons: candidate.reasons,
  source: candidate,
}
```

### 7.4 立直 `riichi`

当前已有 `evaluateRiichiJudgment`，需要升级成真正动作。

合法性：

- 门清。
- 未立直。
- 点数不少于 1000。
- 切牌后听牌。
- 非最后一巡。

生成方式：

```text
对每个切牌候选：
  if candidate.shanten === 0:
    judgment = evaluateRiichiJudgment(handAfterDiscard, candidate, context)
    if judgment.canRiichi:
      生成 riichi 动作候选
```

评分：

```text
score = discardCandidate.score
  + riichiJudgment.score * riichiActionWeight
  + evScore
  - riichiStickCost
```

建议：

- 若 `judgment.shouldRiichi === true`，`riichi` 候选应能压过同切牌的普通 `discard`。
- 若 `judgment.level` 为 `discourage` 或 `strong_discourage`，保留普通切牌，不推荐立直。

EV 接入：

现有 EV action 已支持：

```ts
{ type: "riichi-discard", tile }
```

因此 `riichi` 动作可以先映射到 EV 内部动作：

```ts
estimateRound({
  state,
  action: { type: "riichi-discard", tile },
  candidate,
});
```

### 7.5 碰 `pon`

合法性：

- `state.lastDiscard` 存在。
- 自家手牌中 `lastDiscard.tile` 至少 2 张。
- 自家未立直。
- 碰后至少能切出一张牌。

评估流程：

```text
1. 应用碰动作，生成副露后的手牌和 calls。
2. 对副露后手牌运行 analyzeHandText。
3. 用 open mode 调用 evaluateNanikiru。
4. 每个切牌候选生成一个完整 pon 候选。
5. 与 pass 候选比较。
```

核心评分信号：

- 向听是否下降。
- 是否确定有役。
- 是否获得役牌刻子。
- 是否破坏门清立直价值。
- 副露后打点预估。
- 副露后安全牌数量。
- 需要切出的牌是否危险。

第一版推荐倾向：

- 役牌对子被打出，碰后降向听，积极推荐。
- 断幺确定且能降向听，可以推荐。
- 混一色路线明确且碰后速度提升，可以推荐。
- 无役、两向听以上、破坏门清高价值手，通常不推荐。
- 对手立直时，除非碰后听牌或能明显防守，否则不推荐。

### 7.6 吃 `chi`

合法性：

- 只能吃上家打出的数牌。
- 自家未立直。
- 自家手牌有两张能与 `lastDiscard.tile` 组成顺子。

可能组合：

- `X-2, X-1, X`
- `X-1, X, X+1`
- `X, X+1, X+2`

评估流程同碰：

```text
枚举每一种吃法
  -> 应用副露
  -> 运行副露后切牌分析
  -> 生成 chi + discard 候选
```

核心评分信号：

- 是否确定有役。
- 是否降向听。
- 是否进入听牌。
- 是否终盘需要形式听牌。
- 是否破坏门清立直和高打点路线。
- 吃后切牌危险度。

第一版推荐倾向：

- 断幺确定、吃后降向听，推荐。
- 混一色明确，吃同色牌可推荐。
- 终盘需要抢听牌，可推荐。
- 无役风险、吃后仍慢、吃后必须切危险牌，不推荐。

### 7.7 大明杠 `minkan`

合法性：

- 他家打出牌。
- 自家手牌有同牌 3 张。
- 自家未立直。

评分原则：

- 大明杠非常容易给对手增加宝牌收益，第一版应保守。
- 只有在自家速度、打点和局况都支持时才推荐。

推荐条件：

- 自家听牌或一向听。
- 有确定役或高打点路线。
- 无对手立直等强威胁。
- 当前落后，需要打点。

惩罚：

- 对手立直：强惩罚。
- 自己两向听以上：强惩罚。
- 副露后无役：强惩罚。

### 7.8 暗杠 `ankan`

合法性：

- 自家摸牌后。
- 手牌中存在 4 张相同牌。
- 自家立直后只允许自摸牌补成手内暗刻第四张，且暗杠前后的待牌集合必须完全一致。

评分信号：

- 新宝牌和里宝带来的打点提升。
- 岭上摸牌机会。
- 是否保留门清。
- 是否给对手立直者增加宝牌风险。
- 是否削弱手牌形状或防守资源。

第一版推荐倾向：

- 无人立直、自己一向听以内、杠牌不破坏关键形，可推荐。
- 自家已经听牌且打点不足，可考虑。
- 有对手立直且自己未听牌，通常不推荐。

### 7.9 加杠 `kakan`

合法性：

- 自家已有 `pon`。
- 手牌或摸牌中有第 4 张。
- 自家未立直或规则允许时再判断。

评分：

- 类似暗杠，但比暗杠风险更高。
- 需要考虑抢杠风险，第一版可以作为固定风险惩罚。

推荐倾向：

- 自家听牌、高打点或需要打点时可推荐。
- 对手立直时通常不推荐。
- 终盘领先时不推荐。

### 7.10 跳过 `pass`

合法性：

- 他家打牌后总是合法。

评分：

`pass` 不能只是兜底动作，它是所有鸣牌和荣和候选的对照基线。

评分来源：

- 保持门清立直机会。
- 保持防守牌。
- 避免无役副露。
- 避免吃碰后切危险牌。
- 避免开杠给对手增加宝牌。

第一版实现：

- 对当前手牌运行一次“不鸣后的未来切牌分析”或简化使用当前向听、门清价值和防守余力。
- 如果所有副露候选收益不足，`pass` 应成为推荐。

解释例子：

- `不碰：碰后虽能降向听，但副露后无确定役，且会失去门清立直价值。`
- `不吃：吃后仍是两向听，而且需要切出危险中张。`
- `不杠：已有对手立直，开杠会增加对手宝牌收益。`

## 8. 动作仲裁

不同动作类型的分数不能完全用同一套何切分直接比较。推荐使用分层仲裁：

```text
1. 合法和牌动作
2. 立直 / 切牌 / 杠
3. 副露动作
4. pass
```

但不是简单固定优先级，而是使用类别基础分和策略分：

```text
finalScore =
  categoryBase
  + tacticalScore
  + placementScore
  + evScore
  + reasonPriorityAdjustment
```

建议类别基础分：

| 类别 | base |
| --- | ---: |
| agari | 100000 |
| riichi | 1000 |
| discard | 0 |
| call | 0 |
| kan | -100 |
| pass | 0 |

说明：

- 和牌默认压过其他动作。
- 立直是否压过默听由 `evaluateRiichiJudgment` 和 EV 决定。
- 副露和 `pass` 必须可比较。
- 杠默认保守，必须靠明确收益加回来。

仲裁伪代码：

```ts
export function arbitrateActions(candidates: EvaluatedAction[]): EvaluatedAction | undefined {
  return candidates
    .filter((candidate) => candidate.legal)
    .sort(compareEvaluatedActions)[0];
}
```

排序规则：

```text
1. finalScore 高者优先
2. confidence 高者优先
3. 顺位目标更明确者优先
4. 风险更低者优先
5. 保留原有何切排序作为同分 tie-breaker
```

## 9. EV 扩展

当前 `src/ev/types.ts` 的 `CandidateAction` 已覆盖切牌、立直切牌、副露后切牌和杠动作。EV 仍是快速估算层，不是完整四人搜索。

### 阶段 1：复用现有 EV

支持：

- `discard`
- `riichi`

映射：

```ts
discard -> { type: "discard", tile }
riichi -> { type: "riichi-discard", tile }
```

### 阶段 2：副露后切牌 EV

已支持副露后切牌动作：

```ts
type CandidateAction =
  | { type: "discard"; tile: TileId }
  | { type: "riichi-discard"; tile: TileId }
  | { type: "call-discard"; callType: "chi" | "pon" | "minkan"; calledTile: TileId; tile: TileId };
```

EV 估算时使用“副露后的 state”和“副露后切牌候选”。

### 阶段 3：杠动作 EV

已支持杠动作近似估算：

```ts
| { type: "ankan"; tiles: TileId[] }
| { type: "kakan"; tiles: TileId[] }
```

杠动作 EV 要考虑：

- 新宝牌不确定性。
- 岭上摸牌。
- 对手打点上升。
- 抢杠风险。

当前杠动作 EV 会考虑岭上补一摸、开新宝牌风险和动作类型差异，但仍不是完整杠后搜索。

## 10. 与现有代码的接入

### 10.1 保持公开入口稳定

`chooseAction` 仍是主要入口：

```ts
export function chooseAction(state: GameState, options: ChooseActionOptions = {}): ActionDecision
```

但返回类型需要扩展：

```ts
export interface ActionDecision {
  phase: DecisionPhase;
  mode: StrategyMode;
  action?: DecisionAction;
  candidates: EvaluatedAction[];
  analysis?: EvaluatedNanikiruAnalysis;
  explanation: string;
}
```

为了兼容现有测试和 CLI，可短期保留：

- `analysis`：当推荐链路涉及切牌时返回何切分析。
- `recommendedCandidate`：CLI 中继续输出最佳切牌候选。

### 10.2 切牌链路封装

把当前 `chooseAction` 里的何切逻辑抽成：

```ts
evaluateDiscardDecision(state, options): {
  mode: StrategyMode;
  analysis: EvaluatedNanikiruAnalysis;
  candidates: EvaluatedAction[];
}
```

自摸后普通切牌和副露后切牌都可以复用它。

### 10.3 立直动作升级

当前 `riichi` 已经是独立动作，而不是只改变切牌推荐。

当前行为：

- `discard` 候选保留。
- 如果同一切牌的 `riichiJudgment.shouldRiichi`，额外生成 `riichi` 候选。
- 最终仲裁选择 `riichi` 或 `discard`。

### 10.4 CLI 输出

`decide` CLI 已支持统一动作输出，例如：

```json
{
  "phase": "opponent_discard",
  "mode": "attack",
  "action": {
    "type": "pon",
    "calledTile": "5z",
    "tiles": ["5z", "5z", "5z"],
    "discard": "9m"
  },
  "explanation": "...",
  "candidates": [...]
}
```

轻量 CLI 已支持常见参数：

- `--last-discard <tile>`
- `--last-discard-from left|across|right`

抢杠、岭上和副露后切牌阶段请使用完整 JSON 模式。详见 [decide CLI 文档](./decide-cli.md)。

## 12. 测试计划

### 和牌

- 自摸有役时推荐 `tsumo`。
- 荣和有役且非振听时推荐 `ron`。
- 自家弃过和牌牌时不推荐 `ron`。
- 和牌点数和解释包含役种、番符、点数。

### 立直

- 门清好形听牌推荐 `riichi`。
- 满贯默听且终局领先不推荐立直。
- 低打点窄听且有大量改良时推荐默听改良。
- 对手立直时，低价值听牌不强行追立。

### 吃碰

- 役牌对子被打出，碰后降向听，推荐 `pon`。
- 无役对子被打出，碰后仍慢，推荐 `pass`。
- 上家打出可吃牌，吃后断幺一向听，推荐 `chi`。
- 吃后无役或必须切危险牌，推荐 `pass`。
- 对手立直时，普通吃碰不推荐。

### 杠

- 无威胁且暗杠不破坏手牌，推荐 `ankan`。
- 对手立直时不推荐 `minkan`。
- 终局领先时不推荐加杠。
- 已经可以自摸时优先 `tsumo`，不推荐先杠。

### 兼容

- 原有切牌测试全部通过。
- `npm run decide` 在没有 `lastDiscard` 的场景保持原输出主要字段。
- `--verbose` 能输出所有动作候选。

## 13. 风险和边界

当前仍可暂缓的规则：

- 四杠散了。
- 包牌。
- 副露后一发、岭上、海底、河底等偶发役的完整上下文管理。
- 多人同时可荣和时的结算细节。

第一版必须保证的边界：

- 不推荐非法动作。
- 不因为副露动作破坏现有切牌推荐。
- 不在无役副露场景盲目推荐吃碰。
- 面对立直时，杠和副露要明显更保守。
- 和牌动作必须优先被识别，不能继续推荐切牌。

## 14. 最终目标

完成后，项目的策略入口将从：

```text
自家摸牌后何切推荐
```

升级为：

```text
当前局面动作仲裁器
```

即：

```text
输入 GameState
  -> 识别当前行动阶段
  -> 生成所有合法动作
  -> 对和牌、切牌、立直、副露、杠、不鸣统一评分
  -> 返回推荐动作、候选排序、EV 明细和解释
```

这样后续接入截图识别、HTTP API、MCP 工具或更复杂的顺位策略时，都可以复用同一个动作决策核心，而不是为每种场景单独写分支。
